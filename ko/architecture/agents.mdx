## 에이전트 아키텍처

Decepticon은 **다섯 개의 전문 에이전트**를 통해 작동합니다. 각각은 고유한 도구, 스킬, 클린 컨텍스트 윈도우를 가집니다. 모든 에이전트는 목표마다 새로 스폰됩니다 — 축적된 노이즈 없이, 추론 품질 저하 없이.

## 목표별 클린 컨텍스트

각 에이전트는 모든 목표마다 **새로운 컨텍스트 윈도우**로 스폰됩니다:

```
목표 1 (정찰)         → 새 Recon 에이전트 인스턴스    → 발견 사항 디스크에 저장
목표 2 (익스플로잇)    → 새 Exploit 에이전트 인스턴스   → 목표 1 발견 사항 참조
목표 3 (포스트 익스플) → 새 Post-Exploit 인스턴스      → 목표 1+2 발견 사항 참조
```

- **축적된 노이즈 없음** — 긴 인게이지먼트에서도 컨텍스트가 열화되지 않습니다
- **타겟 컨텍스트** — 각 에이전트는 자신의 목표와 관련된 발견 사항만 수신합니다
- **영속적 상태** — 발견 사항은 에이전트 메모리가 아닌 디스크에 저장됩니다

## 에이전트 미들웨어 스택

각 에이전트는 다음을 제공하는 미들웨어 레이어와 함께 작동합니다:

- **RoE 강제** — 모든 행동이 교전 규칙에 대해 검증됩니다
- **도구 접근** — 영속적 tmux 세션이 있는 샌드박스 실행 환경
- **스킬 카탈로그** — 킬 체인 단계에 기반한 프로그레시브 스킬 디스클로저
- **발견 사항 관리** — 목표, 증거, 권고사항에 대한 구조화된 출력

## 에이전트-도구 상호작용

에이전트는 Kali 샌드박스 내의 영속적 tmux 세션을 통해 오펜시브 도구와 상호작용합니다:

```
에이전트 → Docker 소켓 → Kali 샌드박스 → tmux 세션 → 도구 (sliver-client, nmap 등)
                                              ↓
                                        프롬프트 감지
                                              ↓
                                        후속 명령 전송
```

도구가 대화형 프롬프트(`sliver >`, `msf6 >`, `PS C:\>`)를 표시하면, 에이전트는:
1. 프롬프트 패턴을 감지합니다
2. 도구의 출력을 해석합니다
3. 적절한 후속 명령을 전송합니다
4. 필요 시 제어 시그널(`C-c`, `C-z`)을 처리합니다

이것이 실제 오퍼레이터가 작업하는 방식입니다. Decepticon 에이전트도 동일하게 수행합니다.
